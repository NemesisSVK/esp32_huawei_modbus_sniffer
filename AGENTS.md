# Agent Instructions (ESP32 Huawei Inverter Modbus Sniffer)

## Default workflow preference

- For requests phrased as "analyze", "investigate", "review", "audit", or "what/why/how", default to producing a short findings report (and proposed plan) before making code changes, unless the request explicitly asks to implement/fix/change something immediately.

## Project mission guardrail

- Primary mission: passively decode Modbus RTU traffic between Huawei SUN2000 and meter (Map0-oriented traffic) and publish decoded values.
- The sniffer must remain passive by default: do not introduce RS-485 transmit behavior unless explicitly requested.

## Core architecture guardrails

- Keep frame pipeline intact: UART byte stream -> inter-frame gap boundary -> CRC validation -> request/response correlation -> group-filtered decode -> MQTT publish.
- Keep request/response matching + cold-start fallback behavior in `huawei_decoder`.
- Keep publish-tier model semantics (`TIER_HIGH/MEDIUM/LOW`) and per-group enable behavior consistent with `publish.*` config.
- Keep web stack async-first (`ESPAsyncWebServer` + periodic API polling endpoints).
- Keep MQTT stack async-first (`AsyncMqttTransport` / `MQTTManager`); do not reintroduce blocking client loop patterns.

## Register intelligence guardrails

- Treat `_supporting-projects/huawei-solar-lib-3.0.0/src/huawei_solar/registers.py` as canonical address/type/scale reference for SUN2000 Map0 registers.
- When changing `src/huawei_decoder.cpp` `KNOWN_REGS` entries, cross-check register address, data width, signedness, and scale against canonical register definitions.
- If adding a register not present in canonical `registers.py`, explicitly log it as proprietary/unknown in `DECISIONS.md` and `SESSION_LOG.md`.
- Keep `KNOWN_BLOCKS` fallback signatures aligned with observed live traffic patterns; document any new block signatures with source log references.
- Keep unknown but CRC-valid function codes (`FRAME_UNKNOWN`) as unknown until protocol framing is proven; do not coerce into FC03/FC04 decode paths.

## RS-485 safety rules

- Do not enable active bus driving unless explicitly requested.
- Preserve passive-tap assumptions (auto-control RS-485 board, `rs485_de_re = -1` default behavior).
- Avoid changes that could transmit on bus from sniffer firmware in normal mode.

## Definition of done

- Always report: changed files, behavior impact, and verification commands run.
- Build gates by change scope:
  - Firmware/runtime changes: run `pio run`.
  - OTA-path changes (OTA manager/UI/upload flow): also run `pio run -e esp32-s3-n16r8v-ota`.
  - Settings/schema changes: ensure `build_validate.py` path still passes and keep `config.json.example` synchronized.
- For API/web behavior changes, include a short smoke-check list of touched endpoints/pages.
- For Modbus decode behavior changes, include a short frame/decode sanity checklist (CRC valid request/response pair path, unknown frame handling, group toggle behavior).

## Code date discipline

- This project uses `src/CodeDate.h` as firmware code-date stamp + concise per-release change log.
- When you make firmware/runtime code changes (typically under `src/`, `include/`, `lib/`, or `data/`), update `src/CodeDate.h` in the same session:
  - Set `CODE_DATE` to current UTC in `YYYYMMDD-HHMM` format.
  - Add a new Change log bullet at the top (newest first): `// - YYYYMMDD-HHMM UTC: <short description>`.
  - Ensure the newest Change log timestamp matches `CODE_DATE`.
- `build_validate.py` enforces `CodeDate.h` format/consistency during build; treat failures as incomplete release metadata work.
- Do not update `CODE_DATE` for build-only/flash-only actions when source did not change.

## Generated/build artifacts

- Do not hand-edit generated/build outputs:
  - `src/BuildConfig.h` (generated by `build_validate.py`)
  - anything under `.pio/`, `__pycache__/`, or `compile_commands.json`
- If build flags/includes/dependencies change, regenerate `compile_commands.json` rather than hand-editing.

## Secrets and config hygiene

- Treat these as sensitive: `config.json`, `data/config.json`, `ota.json`, `data/ota.json`.
- Do not print plaintext secrets in logs/chat output.
- Keep `config.json` and `data/config.json` strict JSON (no comments).
- Keep `config.json.example`/`ota.json.example` sanitized and documentation-friendly.

## Settings/schema consistency

- When settings schema changes, update together:
  - `src/ConfigManager.h` / `src/ConfigManager.cpp`
  - `build_validate.py`
  - `config.json.example`
  - relevant UI handlers in `src/web_ui.cpp`

## Web/API contract guardrails

- Keep existing API shapes/status semantics stable unless explicitly requested.
- Preserve explicit OTA arming flow (`POST /api/ota_arm`) and status polling (`GET /api/ota_status`).
- Preserve dashboard polling endpoints (`/api/status`, `/api/values`, monitoring/log endpoints) as lightweight operations.
- Keep auth + IP whitelist checks consistent across protected endpoints.

## Logging conventions

- Use `UnifiedLogger` for runtime logs (`info`, `warning`, `error`; `verbose` only when needed).
- Avoid high-frequency spam logs in sniffer hot path unless specifically debugging.
- Never log plaintext secrets.

## Supporting-project references

- `_supporting-projects/` is reference context (HA manager + huawei_solar integrations/libs).
- Treat supporting projects as read-only reference unless explicitly asked to modify them.

## AI memory maintenance (required)

- Treat these files as the project memory system:
  - `AI_CONTEXT.md`, `TASKS.md`, `DECISIONS.md`, `HISTORY.md`, `SESSION_LOG.md`
- When project state changes (code/config/contracts/workflow), update memory files in the same session:
  - `TASKS.md`: move/update task status and next action.
  - `SESSION_LOG.md`: append one new UTC entry per work session.
  - `DECISIONS.md`: add durable decision entries only.
  - `AI_CONTEXT.md`: update stable architecture/contracts/build-context when changed.
  - `HISTORY.md`: add milestone entries for meaningful evolution points.
- Validation gate for memory consistency:
  - Run `powershell -ExecutionPolicy Bypass -File scripts/validate_ai_memory.ps1` after memory updates.
  - Treat validator errors as incomplete documentation work.
  - Validator freshness rule: memory is stale if relevant project files are newer than latest `S-*` UTC entry in `SESSION_LOG.md`.
- MCP augmentation policy:
  - Repository handoff files remain the primary source of truth; MCP memory is a secondary acceleration layer.
  - Use `mcp__memory__` to store/retrieve durable facts (invariants, stable contracts, accepted decisions, recurring risks).
  - Do not store secrets, transient chat text, or raw command outputs in MCP memory.
  - If MCP memory is unavailable, continue normally with file-based memory maintenance only.
  - Use `mcp__sequentialthinking__` only for complex debugging/design tradeoffs; do not require it for routine edits.
  - Use external research MCPs (`context7`, `tavily`, etc.) only when freshness/official-reference needs justify them.
  - After MCP-memory-relevant updates, run `powershell -ExecutionPolicy Bypass -File scripts/update_mcp_sync_state.ps1 -SessionId <latest S-id>` to refresh `MCP_MEMORY_SYNC.json`.
  - Consider MCP sync incomplete until both `update_mcp_sync_state.ps1` and `validate_ai_memory.ps1` pass.
- Keep entries concise and high-signal; avoid transcript dumps.

## Quick sanity checks

- After firmware changes: `pio run`
- After OTA-path changes: `pio run -e esp32-s3-n16r8v-ota`
- After settings/schema changes: verify validation flow + examples consistency
